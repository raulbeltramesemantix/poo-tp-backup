# Questao 18

Pedido do professor:

Instanciar um iterator para a ArrayList de pessoas, funcionario ou cliente, percorrer com while(iterator.hasNext()) e imprimir iterator.next(). Explicar como acontece, relacionar com foreach e testar foreach.

O que foi feito:

No arquivo TestesQuestoesSistema.java foi criado o bloco:

// questao 18

Dentro dele eu criei um CadastroUsuarios, cadastrei 3 funcionarios e copiei a lista para uma ArrayList<Funcionario>.

Depois foi criado:

```java
Iterator<Funcionario> iterator = funcionarios.iterator();
```

E a lista foi percorrida assim:

```java
while (iterator.hasNext()) {
    System.out.println(iterator.next());
}
```

Depois foi feito o mesmo teste usando foreach:

```java
for (Funcionario funcionario : funcionarios) {
    System.out.println(funcionario);
}
```

Como explicar:

O Iterator funciona como um cursor que passa pela lista. O metodo hasNext() pergunta se ainda existe um proximo elemento. O metodo next() devolve esse elemento e avanca o cursor para a proxima posicao.

O foreach em Java e uma forma mais curta de fazer essa mesma ideia. Quando usamos:

for (Funcionario funcionario : funcionarios)

o Java usa internamente o mecanismo de Iterable/Iterator da colecao. Por isso o foreach e uma forma simplificada e mais legivel de percorrer a lista.

Como mostrar:

Rodar TestesQuestoesSistema.java e mostrar no console a parte:

- q18 percorrendo com Iterator
- q18 percorrendo com foreach

As duas partes imprimem os mesmos funcionarios.


# Questao 19

Pedido do professor:

Apresentar no main testes do comparator implementado. Usar Collections.sort passando o comparator criado para ordenar a lista com dois parametros diferentes. Rodar duas vezes.

O que ja existia:

A classe Cliente ja implementava Comparator<Cliente>. O metodo compare da classe Cliente compara pelo idCliente:

```java
public int compare(Cliente primeiroCliente, Cliente segundoCliente) {
    return Integer.compare(primeiroCliente.getIdCliente(), segundoCliente.getIdCliente());
}
```

A classe Cliente tambem possui um comparator por nome:

```java

```

O que foi feito:

# Questao 20

Pedido do professor:

Criar um metodo find para clientes utilizando Iterator e Comparator. Apresentar testes do metodo implementado. Fazer chamadas ao binarySearch() da classe Collections e comparar com o find implementado.

O que foi feito:

No arquivo CadastroClientes.java foi criado o metodo:

```java
public Cliente find(Cliente clienteProcurado, Comparator<Cliente> comparator) {
    Iterator<Cliente> iterator = getClientes().iterator();
    while (iterator.hasNext()) {
        Cliente cliente = iterator.next();
        if (comparator.compare(cliente, clienteProcurado) == 0) {
            return cliente;
        }
    }
    return null;
}
```

Esse metodo percorre a lista com Iterator e usa o Comparator recebido para saber se o cliente atual e igual ao cliente procurado.

No arquivo TestesQuestoesSistema.java foi criado o bloco:

// questao 20

Foram feitos dois testes.

Teste 1: busca por idCliente.

Foi criado:

```java
Comparator<Cliente> comparadorPorId = new Cliente();
Cliente clienteProcuradoPorId = new Cliente(2, "", "", "", 0, null);
```

Depois foi chamado:

```java
cadastroClientes.find(clienteProcuradoPorId, comparadorPorId);
```

Depois a lista foi ordenada com o mesmo comparator:

```java
Collections.sort(clientes, comparadorPorId);
```

E foi chamado:

```java
Collections.binarySearch(clientes, clienteProcuradoPorId, comparadorPorId);
```

Teste 2: busca por nome.

Foi criado:

```java
Comparator<Cliente> comparadorPorNome = Cliente.porNome();
Cliente clienteProcuradoPorNome = new Cliente(0, "Bruno", "", "", 0, null);
```

Depois foi chamado find com esse comparator.

Depois a lista foi ordenada por nome e foi chamado binarySearch com o mesmo comparator.

Como explicar:

O find implementado faz busca linear. Ele percorre do inicio ao fim usando Iterator. Ele nao precisa que a lista esteja ordenada.

O binarySearch faz busca binaria. Ele e mais rapido em listas grandes, mas so funciona corretamente se a lista estiver ordenada pelo mesmo Comparator usado na busca.

Outra diferenca importante:

O find retorna o Cliente encontrado ou null.

O binarySearch retorna o indice onde o Cliente foi encontrado. Se nao encontrar, retorna um numero negativo.

Como mostrar:

Rodar TestesQuestoesSistema.java e mostrar no console:

- q20 find por idCliente usando Iterator e Comparator
- q20 binarySearch por idCliente apos ordenar com o mesmo Comparator
- q20 find por nome usando Iterator e Comparator
- q20 binarySearch por nome apos ordenar com o mesmo Comparator

Nos testes feitos, o find e o binarySearch encontram o mesmo cliente.

# Como falar a Questao 21 na apresentacao

Voce pode explicar assim:

Eu preparei o sistema cadastrando ingredientes, produtos com receita, motoqueiros e regioes. Depois cadastrei 10 clientes com enderecos diferentes. Em seguida entrei como funcionario e criei pedidos para esses clientes, variando quantidade, produto, pedidos com mais de um item, pedidos cancelados e pedidos com entrega.

Quando registro a venda, o sistema gera um recibo e baixa automaticamente o estoque conforme a receita do produto vendido. Essa baixa acontece em ControleEstoque.atualizarAposPedido.

Para entrega, eu uso a opcao informar entrega via WhatsApp. O sistema associa uma entrega a venda, escolhe um motoqueiro ativo e define a regiao pelo endereco do cliente. Em alguns casos eu confirmo a entrega, em outros deixo em rota. Tambem testo um endereco com bairro desconhecido para mostrar a regra de regiao padrao.

Para cancelamento, uso a opcao cancelar pedido antes de registrar a venda. O sistema altera o status do pedido para cancelado e mostra a retencao.

No final, mostro o estoque atualizado, os recibos das vendas, os relatorios/resumo do sistema e salvo os dados.
